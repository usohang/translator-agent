# 문서 번역 에이전트 — 기획서 (Build Spec)

> 이 문서는 **Claude Code에 그대로 입력하기 위한 빌드 스펙**입니다.
> 구현 시 이 문서의 "데이터 모델", "파이프라인", "출력 규칙"을 그대로 따르면 됩니다.

---

## 1. 프로젝트 개요

### 1.1 한 줄 정의
PDF·HWP 등 문서 파일을 업로드하면, **사람 번역가 수준의 문맥 번역**으로 한글 문서를 새로 생성해 돌려주는 AI 번역 에이전트. 웹 대시보드로 업로드·진행상황·다운로드를 관리한다.

### 1.2 핵심 가치 (차별점)
- **단순 기계 번역이 아님**: 문맥·도메인·용어 일관성을 지키는 "번역가형" 번역
- **레이아웃 보존**: 제목/소제목 계층, 표, 그림, 문서 디자인을 가능한 한 원문과 동일하게 재현
- **원문 추적성**: 번역본을 보다가 원문 몇 페이지인지 즉시 찾을 수 있는 페이지 매핑
- **시각화**: 웹 대시보드에서 처리 과정과 결과를 한눈에 확인

### 1.3 입력 / 출력
- **입력**: `.pdf`, `.hwp`, `.hwpx`, `.docx` (1차 범위). 추후 `.pptx`, 이미지(OCR) 확장
- **출력**: 번역된 `.docx` (기본 권장) — 편집·재활용이 쉽고 스타일 제어가 정밀함
  - 옵션: `.pdf` (레이아웃 고정이 필요한 경우 DOCX→PDF 변환)

## 2. 기능 요구사항 (정리)

| # | 요구사항 | 상세 |
|---|---------|------|
| F1 | 문서 번역 | 원문 → 한글, 문맥 기반 자연스러운 번역 |
| F2 | 본문 글씨 | 기본 **11pt** |
| F3 | 제목/소제목 강조 | 계층(H1~H3)에 따라 글자 크기 자동 확대 + 굵게 |
| F4 | 표 보존 | 표 구조·셀 내용 번역 후 재현 (가능 범위까지) |
| F5 | 그림 보존 | 원문 이미지 추출 후 동일 위치에 재삽입 |
| F6 | 디자인 보존 | 가능한 선까지 재현, 어려우면 단순화 (degrade gracefully) |
| F7 | 페이지 매핑 | 매 페이지/챕터마다 `[원문 p.N]` 표기 + 전체 매핑 표 제공 |
| F8 | 웹 대시보드 | 업로드·진행률·미리보기·다운로드 |
| F9 | 용어 일관성 | 용어집(glossary) 기반 동일 용어 동일 번역 |

## 3. 시스템 아키텍처

```
[웹 대시보드 (Next.js)]
        │  업로드 / 진행률(SSE) / 다운로드
        ▼
[API 서버 (FastAPI)]
        │  Job 생성
        ▼
[작업 큐 (Celery/RQ + Redis)]
        ▼
┌─────────────────────────────────────────────┐
│           번역 파이프라인 (워커)               │
│                                               │
│  1. Ingest  →  2. Parse  →  3. IR 생성        │
│       →  4. Translate (Claude API)            │
│       →  5. Reassemble  →  6. Output          │
└─────────────────────────────────────────────┘
        │
        ▼
[저장소 (로컬 / S3)]  ──  원문 / IR / 번역본 / 매핑표
```

핵심 설계 원칙: **모든 포맷을 공통 중간표현(IR, Intermediate Representation)으로 정규화**한 뒤 번역·재조립한다. 그래야 입력 포맷이 늘어나도 번역/조립 로직을 재사용할 수 있다.

## 4. 처리 파이프라인 (단계별 상세)

### Step 1. Ingest (수집)
- 업로드 파일 검증: 확장자, 크기, 암호화 여부, 손상 여부
- 메타데이터 추출(페이지 수, 언어 감지)

### Step 2. Parse (구조 추출)
포맷별 파서로 **텍스트 + 레이아웃 + 표 + 이미지 + 페이지 경계**를 추출. (포맷별 전략은 5장 참고)

### Step 3. IR 생성 (정규화)
추출 결과를 공통 문서 모델로 변환. (데이터 모델은 9장 참고)
- 각 블록에 `type`, `original_page`, `style_hint` 부여
- 제목 계층 추론 (폰트 크기/굵기/번호 패턴 기반 → H1/H2/H3)

### Step 4. Translate (번역) — *핵심*
- **블록 단위 번역 + 문서 전체 컨텍스트 주입** (앞뒤 맥락, 용어집, 문서 요약)
- 번역 대상이 아닌 것 보호: 숫자, 코드, 수식, URL, 고유명사(선택), 표의 단위
- 번역가 페르소나 시스템 프롬프트 + 도메인 자동 감지 (법률/기술/의료/일반 등)
- **2-pass**: ① 초벌 번역 → ② 검수 패스(자연스러움·일관성·누락 점검)
- 용어집에 신규 용어 자동 등록 → 같은 문서 내 일관성 유지

### Step 5. Reassemble (재조립)
- IR(번역 완료) → 출력 포맷으로 빌드
- 스타일 규칙 적용 (8장): 본문 11pt, 제목 확대, 표/이미지 재배치
- 페이지 매핑 표기 삽입 (7장)

### Step 6. Output (산출)
- `.docx` 생성 → (옵션) `.pdf` 변환
- 원문-번역본 페이지 매핑 표를 문서 끝 또는 별도 시트로 첨부

## 5. 파일 포맷별 처리 전략

### 5.1 PDF
| 방법 | 도구 | 비고 |
|------|------|------|
| 레이아웃 인식 추출 (권장) | **Docling** (IBM) | 표/제목/구조를 인식해 Markdown·JSON으로 출력. 고품질 |
| 좌표 기반 추출 | **PyMuPDF (fitz)** | 텍스트 위치·폰트·이미지 추출에 강력 |
| 표 정밀 추출 | **pdfplumber** | 표 경계 추출 보조 |
| 스캔 PDF | OCR (Tesseract / PaddleOCR) | 이미지형 PDF 대응 (2차 확장) |

> 권장 조합: **Docling(구조) + PyMuPDF(이미지·정밀 좌표)** 병행.

### 5.2 HWP / HWPX (한글 — 난이도 높음)
HWP는 한컴의 독자 바이너리(OLE2+zlib) 포맷이라 완벽 파싱이 어렵다. 우선순위로 접근:

1. **`hwp-hwpx-parser`** (순수 파이썬, 권장 1순위)
   - 텍스트, **표(markdown/CSV)**, **이미지**, 각주/메모 추출 지원
   - HWP5·HWPX 모두 대응, JVM 불필요
2. **`pyhwp`** — HWP5 파서. `hwp5html`로 HTML 변환 가능(레이아웃 일부 보존), 이미지 추출(`hwp5proc cat ... BinData/...`)
3. **`hwplib-py`** — 표/이미지 컨트롤 접근, JSON 익스포트
4. **LibreOffice headless + H2Orestart 확장** (대안)
   - `soffice --headless --convert-to docx file.hwp` 로 HWP→DOCX 변환 후 DOCX 파이프라인 재사용
   - 복잡한 레이아웃을 한 번에 처리할 수 있어 fallback으로 유용

> **전략**: HWPX(=XML 기반, 최신 한글) 우선 처리가 가장 깔끔. 구형 HWP는 `hwp-hwpx-parser`로 콘텐츠 추출 → 실패 시 LibreOffice 변환 fallback.

### 5.3 DOCX
- **`python-docx`** 로 직접 파싱·생성. 입력·출력 모두 가장 안정적 → 모든 포맷의 "공통 출력 백엔드"로 사용 권장.

## 6. 번역 품질 전략 (가장 중요한 차별점)

### 6.1 번역 엔진
- **Anthropic Claude API** 사용 (Claude Code로 빌드하므로 자연스러운 선택)
- 모델: 품질 우선 작업엔 상위 모델, 대량/단순 블록엔 경량 모델로 비용 최적화 (라우팅)

### 6.2 품질 확보 기법
- **컨텍스트 주입**: 문서 전체 요약 + 직전/직후 블록을 함께 전달 → 대명사·맥락 유지
- **용어집(Glossary)**: 문서별/사용자별 용어 사전. "동일 용어 = 동일 번역" 강제
- **번역 메모리(TM)**: 과거 번역 재사용으로 일관성·속도 향상 (2차)
- **도메인 감지**: 문서 성격(계약서/논문/매뉴얼 등) 자동 분류 → 톤·문체 조정
- **검수 패스**: 초벌 후 "원문 의미 누락/오역/어색함" 자체 점검 프롬프트
- **보호 토큰**: 번역하면 안 되는 요소를 플레이스홀더로 치환 후 복원
  - 예: `{{NUM_01}}`, `{{CODE_02}}`, `{{URL_03}}`

### 6.3 번역 프롬프트 설계 가이드 (예시 골격)
```
당신은 [도메인] 전문 번역가입니다.
- 원문의 의미·뉘앙스·논조를 보존하되 자연스러운 한국어로 옮깁니다.
- 아래 용어집을 반드시 따릅니다: {glossary}
- {{...}} 형태의 토큰은 번역하지 말고 그대로 둡니다.
- 표/목록 구조는 유지합니다.
[문서 요약]: {summary}
[직전 맥락]: {prev}
[번역할 블록]: {block}
```

## 7. 원문 ↔ 번역본 페이지 매핑 설계 (F7)

번역하면 분량이 달라져 페이지가 어긋난다. 추적성을 위해:

1. **블록마다 `original_page` 기록** (Parse 단계에서 확보)
2. **본문 내 표기**: 챕터/섹션 시작 지점 또는 원문 페이지가 바뀌는 지점에 표식 삽입
   - 예: 소제목 옆 또는 머리말에 `〔원문 p.12〕`
   - 표기 스타일은 회색·작은 글씨로 본문 가독성 방해 최소화
3. **매핑 표 첨부**: 문서 끝에 대조표 생성

   | 번역본 페이지 | 원문 페이지 | 챕터/섹션 |
   |---|---|---|
   | 1–2 | 1 | 서론 |
   | 3 | 2–3 | 1. 배경 |

4. (대시보드 옵션) **좌우 동기화 뷰**: 번역본 클릭 시 해당 원문 위치로 점프

## 8. 출력 포맷 & 스타일 규칙 (F2~F6)

### 8.1 글꼴 / 크기
| 요소 | 스타일 |
|------|--------|
| 본문 | **11pt**, 일반 |
| H1 (대제목) | 20–24pt, Bold |
| H2 (중제목) | 16–18pt, Bold |
| H3 (소제목) | 13–14pt, Bold |
| 캡션/주석 | 9pt, 회색 |
| 페이지 표기 | 9pt, 회색 |

- 한글 기본 글꼴: 본문 가독성 좋은 글꼴(예: 맑은 고딕/나눔고딕 계열) 지정
- 제목 크기는 **계층에 따라 자동 산정**(본문 대비 배율)

### 8.2 표 / 그림
- 표: 행·열 구조 유지, 셀 텍스트만 번역, 헤더 강조 유지
- 그림: 원본 이미지 비트맵 그대로 재삽입, 캡션은 번역
- **그래이스풀 디그레이드**: 복잡한 도형/텍스트박스 등 완벽 재현이 어려우면
  - ① 가능한 근사치로 배치 → ② 그래도 어려우면 "원문 이미지 캡처"로 삽입 + 번역 캡션 병기

### 8.3 출력 백엔드
- **1차: `python-docx`** 로 DOCX 직접 생성 (스타일 제어 정밀)
- **변환: `mcp-pandoc`** 또는 LibreOffice로 DOCX→PDF (옵션)

## 9. 데이터 모델 (IR — Intermediate Representation)

```jsonc
{
  "doc_id": "uuid",
  "source_format": "pdf | hwp | hwpx | docx",
  "language_src": "en",
  "language_tgt": "ko",
  "glossary": [{ "src": "term", "tgt": "용어" }],
  "blocks": [
    {
      "id": "b001",
      "type": "heading | paragraph | table | image | caption | list",
      "level": 1,                 // heading일 때만 (1~3)
      "original_page": 12,        // 원문 페이지 (매핑용)
      "order": 37,                // 문서 내 순서
      "text_src": "원문 텍스트",
      "text_tgt": "번역 텍스트",   // Translate 단계에서 채움
      "style_hint": {             // Parse가 추출한 스타일 단서
        "font_size": 11,
        "bold": false,
        "align": "left"
      },
      "table": {                  // type=table일 때
        "rows": [["셀", "셀"], ["...", "..."]]
      },
      "image_ref": "img/BIN0002.png"  // type=image일 때
    }
  ],
  "page_map": [
    { "tgt_page": "1-2", "src_page": "1", "section": "서론" }
  ]
}
```

## 10. 웹 대시보드 설계 (F8)

### 10.1 화면 구성
1. **업로드**: 드래그&드롭, 다중 파일, 포맷 표시
2. **작업 목록**: 진행 중/완료 잡, 상태 배지
3. **진행 상황(실시간)**: 파싱 → 번역 → 조립 단계별 프로그레스 바 (SSE/WebSocket)
4. **미리보기**: 원문/번역본 좌우 비교 뷰 + 페이지 매핑 점프
5. **다운로드**: DOCX/PDF, 매핑표
6. **설정**: 용어집 관리, 출력 포맷, 모델 선택

### 10.2 기술
- 프론트: **Next.js + React + Tailwind CSS**, 진행률은 SSE
- 상태: 업로드→QUEUED→PARSING→TRANSLATING→ASSEMBLING→DONE/ERROR
- 디자인: 좌측 잡 리스트 + 우측 상세, 깔끔한 단색 계열, 단계별 시각화 카드

## 12. 추천 MCP / Skill

### 12.1 Claude Code에서 활용할 Skill (Anthropic 공식)
이미 검증된 문서 생성 스킬을 그대로 쓰면 출력 품질이 올라간다.
- **docx skill** — 번역 결과 Word 문서 생성/스타일링 (본 프로젝트 출력 핵심)
- **pdf skill** — PDF 생성·폼·병합·워터마크
- **pptx / xlsx skill** — 향후 슬라이드/스프레드시트 확장 시
- **frontend-design skill** — 대시보드 UI 설계 시

### 12.2 MCP 서버
- **mcp-pandoc** — 포맷 간 변환(MD/HTML/DOCX/PDF/LaTeX 등). 출력 파이프라인 보조
- **Filesystem MCP** — 업로드/산출 파일 접근
- (자체 제작 권장) **HWP MCP** — `hwp-hwpx-parser`를 감싸 "HWP 추출" 툴을 MCP로 노출하면 에이전트가 직접 호출 가능

> 참고: `mcp-pandoc`은 PDF 출력 시 TeX Live 설치가 필요할 수 있음. HWP 직접 변환은 미지원이라 5.2의 파서/LibreOffice 경로를 사용.

## 13. 개발 로드맵 (마일스톤)

### M1 — MVP (핵심 흐름 1개 포맷)
- PDF → IR → 번역 → DOCX 출력 (11pt 본문 + 제목 계층)
- CLI 또는 최소 업로드 UI
- 페이지 매핑 표기 기본형

### M2 — 품질 & 포맷 확장
- 표/이미지 보존
- HWP/HWPX 지원 추가
- 용어집 + 2-pass 검수
- 페이지 매핑 표 + 좌우 비교 뷰

### M3 — 대시보드 완성
- 실시간 진행률, 잡 관리, 미리보기, 다운로드
- PDF 출력 옵션, 모델 라우팅(비용 최적화)

### M4 — 고도화 (선택)
- 스캔 PDF OCR, PPTX 지원, 번역 메모리(TM), 사용자별 용어집 저장

## 14. 리스크 & 고려사항

| 리스크 | 대응 |
|--------|------|
| HWP 완벽 파싱 어려움 | 파서 1순위 + LibreOffice fallback + 최후엔 이미지+캡션 |
| 복잡한 레이아웃 100% 재현 불가 | "가능 범위까지" 원칙 + graceful degrade 명시 |
| 번역 비용/속도 | 블록 배치 처리, 모델 라우팅, 캐싱(TM) |
| 페이지 매핑 정확도 | Parse 단계에서 `original_page`를 모든 블록에 확실히 부여 |
| 긴 문서 컨텍스트 한계 | 문서 요약 + 슬라이딩 윈도우로 맥락 유지 |
| 보안/개인정보 | 업로드 파일 보존기간·삭제 정책, 암호화 |

## 15. Claude Code 시작 가이드

Claude Code에 이 문서를 주고 다음 순서로 진행하도록 지시하면 효율적이다.

1. **M1부터**: "9장 데이터 모델(IR)을 먼저 정의하고, PDF→IR→DOCX 최소 파이프라인을 만들자."
2. **파서 검증**: 실제 샘플 PDF/HWP로 추출 품질부터 확인 (번역은 그 다음)
3. **출력 스타일**: 8장 표를 python-docx 스타일 규칙으로 구현
4. **번역 모듈**: 6장 프롬프트 골격 + 보호 토큰 + 용어집부터
5. **페이지 매핑**: 7장 — 블록의 `original_page`를 산출물에 반영
6. **대시보드**: 마지막에 FastAPI + Next.js로 감싸기

### 권장 디렉터리 구조 (제안)
```
translator-agent/
├── backend/
│   ├── parsers/        # pdf.py, hwp.py, docx.py → IR
│   ├── ir/             # 데이터 모델 정의
│   ├── translate/      # Claude 호출, 용어집, 검수
│   ├── assemble/       # IR → docx/pdf, 스타일 규칙
│   ├── pagemap/        # 원문-번역본 매핑
│   ├── jobs/           # 큐/워커
│   └── api/            # FastAPI 엔드포인트
├── frontend/           # Next.js 대시보드
├── samples/            # 테스트 문서
└── README.md
```
